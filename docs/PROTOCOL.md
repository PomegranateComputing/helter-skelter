# Bridge <-> Orchestrator Protocol

Versioned message protocol between `bridge/openrct2-plugin` (TypeScript,
runs inside OpenRCT2) and `core/orchestrator` (Rust, runs the observe →
propose → authorize → execute → log loop from `docs/VISION.md`). This
document is the transport and compatibility policy; the wire shape itself
is defined in `bridge/protocol/*.schema.json` and must stay in sync with the
Rust types in `core/common/src/protocol/` and the TS types in
`bridge/openrct2-plugin/src/protocol/` — see "Keeping schema and types in
sync" below.

## Transport (0.1)

- **Localhost TCP**, one connection per running simulation. The bridge
  plugin is the TCP client; the orchestrator is the TCP server (it's the
  long-lived process; the plugin comes and goes with the game).
- **Newline-delimited JSON (NDJSON)**: each message is one JSON object
  (an `Envelope`, see below) serialized on a single line, terminated by
  `\n`. No length-prefixing, no framing beyond the newline, in 0.1.
- Both sides send `hello` immediately on connect and expect one back
  before sending anything else. Either side may send `heartbeat` on an
  interval; the receiver uses missed heartbeats (not the OS TCP timeout)
  to detect a stalled peer, per the watchdog requirement in
  `docs/VISION.md`.
- **Protobuf is a later candidate**, not a 0.1 commitment. NDJSON was
  chosen for 0.1 because it's trivially inspectable in logs and via `nc`/
  `tcpdump` while the protocol itself is still changing shape; Protobuf's
  binary framing and schema-compiler step are worth it once the message
  set stabilizes (likely around 0.2, once the architect's terrain/zoning
  messages exist) but add friction now for no runtime benefit at
  localhost latencies.

## Envelope

Every message is a single JSON object with these fields (see
`bridge/protocol/envelope.schema.json` for the normative schema):

| Field | Type | Notes |
|---|---|---|
| `protocol_version` | `"0.1.0"` | Const for this milestone. |
| `message_id` | UUIDv7 | Generated by the sender; time-ordered. |
| `timestamp` | RFC3339 string | Wall-clock send time. |
| `simulation_id` | UUID | Stable per running simulation. |
| `kind` | enum | See "Message kinds" below. |
| `payload` | object | Shape depends on `kind`. |
| `correlation_id` | UUID or `null` | `message_id` this message responds to. Mandatory (non-null) for `command.result` **and `ack`** — an ack or result with no correlation_id would be structurally meaningless, so this milestone requires it even though only `command.result` was called out explicitly when this protocol was designed (ADR-0002). |
| `status` | `"ok"` / `"error"` / `null` | Set on results; `null` on requests/events. |
| `error` | structured `{code, message}` or `null` | Never free-form text where structure fits. |

`message_id` is UUIDv7 specifically (not just any UUID) so that logs and
the decision ledger sort in send order without a secondary timestamp
comparison. `simulation_id` and `correlation_id` are validated as UUIDs
generally, not v7 specifically, since `correlation_id` is just a copy of a
prior `message_id` and doesn't need its own generation constraint.

## Message kinds

- `hello` — first message on a fresh connection, either direction. Carries
  `role` (`bridge`/`orchestrator`), `bridge_version`, `openrct2_version`.
- `heartbeat` — carries a `tick` (monotonic per connection; the schema
  can't express monotonicity across messages, the orchestrator enforces
  it).
- `observation.snapshot` — full park-state export: `park_date`, `cash`,
  `loan`, `park_rating`, `guest_count`, `rides[]`, `staff_counts`,
  `weather`. `park_date` is a structured `{year, month, day}` rather than
  an ISO calendar date because OpenRCT2's in-game calendar has **8 months
  per year**, not the Gregorian 12 — an ISO date would misrepresent it.
  `ride.status` and `weather`'s exact enum values are 0.1 placeholders
  pending verification against the real OpenRCT2 plugin API once
  `bridge/openrct2-plugin` actually calls it (it's still a stub); expect
  these two enums specifically to be revised without a protocol version
  bump, since they describe engine facts, not wire-format shape.
- `command.request` — a bounded action proposal (see below). Carries
  `action`, `params` (shape depends on `action`), `idempotency_key` (the
  orchestrator must treat repeated delivery of the same key as a no-op
  after first execution), `expiry_tick`.
- `command.result` — the outcome of a `command.request`. Carries
  `engine_cost` (nullable) and `engine_error` (nullable, structured).
  `correlation_id` on the envelope is mandatory and points at the
  originating `command.request`'s `message_id`.
- `shutdown` — either side announcing it's about to close the connection,
  with a `reason` (`operator_request` / `fatal_error` / `watchdog_timeout`).
- `ack` — acknowledges receipt of a message by `correlation_id`. Empty
  payload; the correlation_id *is* the content.

## Bounded actions (0.1)

Exactly five `command.request` actions exist in this milestone — see
`bridge/protocol/commands/*.schema.json`:

- `set_ride_price { ride_id, price }`
- `set_park_entry_fee { price }`
- `hire_staff { type }` (`type` ∈ `handyman`/`mechanic`/`security`/`entertainer`)
- `open_ride { ride_id }`
- `close_ride { ride_id }`

Adding a sixth action means adding a schema, a fixture, a Rust
`CommandAction` variant, and a TS `CommandAction` union member — not just
one of those. All four are checked against each other by the sync tests
described below, so an incomplete addition fails CI rather than silently
drifting.

## Design note: no cross-file `$ref` in the schemas

`bridge/protocol/envelope.schema.json` intentionally validates `payload`
only as `{"type": "object"}`, and `messages/command_request.schema.json`
validates `params` the same way, rather than using JSON Schema `$ref`
across files to pin the exact payload/params shape to `kind`/`action`.
Every `.schema.json` file in this repo is independently loadable and
compilable with zero resolver logic.

The kind ↔ payload (and action ↔ params) correspondence is instead
enforced twice, redundantly:

1. **By construction**, in both languages: Rust's `Payload`/`CommandAction`
   are serde adjacently-tagged enums, and TypeScript's `Payload`/
   `CommandAction` are discriminated unions — in both, it is a compile
   error to pair the wrong payload with a `kind`.
2. **By the test harness**, in both languages: given a fixture's `kind`
   (and, for `command.request`, `action`), the test looks up and validates
   against the correct `messages/*.schema.json` (and `commands/*.schema.json`)
   file directly — see "Keeping schema and types in sync" below.

This was a deliberate simplification over `$ref`-based composition, not an
oversight: it avoids needing a custom schema resolver (both `ajv` and the
Rust `jsonschema` crate resolve `$ref` by URI, which either requires
network-shaped fake `$id`s and a custom in-memory resolver, or doesn't work
across files at all without one) for a milestone where the message set is
still small and enumerable by hand in the test harness.

## Keeping schema and types in sync

`bridge/messages/fixtures/*.json` is the single source of truth: one
fixture per message kind, plus one `command.request` fixture per bounded
action and one `command.result` fixture for both the ok and error case
(12 total). Both languages run the same check against the same fixtures:

- **Rust** (`core/common/tests/protocol_roundtrip.rs`): for every fixture,
  validate it against `envelope.schema.json`, validate `payload` against
  the matching `messages/*.schema.json`, validate `params` against the
  matching `commands/*.schema.json` for `command.request`, deserialize
  into `common::protocol::Envelope`, run `Envelope::validate()`, and
  re-serialize — asserting the result equals the original fixture
  byte-for-byte (as a `serde_json::Value`).
- **TypeScript** (`bridge/openrct2-plugin/test/protocol_roundtrip.test.js`):
  the same validate-against-schema steps using `ajv` (draft 2020-12), then
  round-trip each fixture through `decodeEnvelope`/`encodeEnvelope`.

  Note on what the TS side *doesn't* prove: an earlier draft of this test
  tried to get a compile-time check by statically importing the JSON
  fixtures and asserting `satisfies Envelope` against them. That doesn't
  work — TypeScript widens string-literal fields on JSON module imports
  (e.g. `kind: "hello"` becomes `kind: string`), so a discriminated union
  keyed on string literals can never structurally match an imported JSON
  value, compile error or not. The runtime `ajv` validation is the real
  check for TS; the type definitions in `src/protocol/types.ts` are
  developer-facing documentation checked only by ordinary `tsc` usage
  elsewhere in the plugin, not by this test.

Both suites run in the existing CI jobs (`.github/workflows/ci.yml`'s
`rust` and `bridge` jobs) — there's no separate third "sync check" job;
`cargo test` and `pnpm test` failing on a schema/fixture/type mismatch
*is* the sync check.

## Versioning and compatibility policy

- `protocol_version` is a plain string const (`"0.1.0"` for this
  milestone), not semver-parsed at runtime in 0.1 — there's only one
  version in flight.
- A **backward-compatible** change (new optional field, new enum value
  additive in a non-breaking position, new message kind) increments the
  patch/minor version and both sides accept the old and new value during
  a transition; a receiver must never fail on an unrecognized field.
- A **breaking** change (removing/renaming a required field, changing a
  field's type, removing an enum value, changing an existing action's
  params shape) requires a new `protocol_version` and both sides reject a
  mismatched version at `hello` rather than guess. There is no
  cross-version negotiation in 0.1 — bridge and orchestrator are expected
  to be redeployed together.
- Every wire-shape change lands with: the schema file(s), the fixture(s),
  both language's types, and both round-trip tests, in the same commit —
  see `docs/CODING_STANDARD.md`.
